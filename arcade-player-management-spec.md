# Arcade Player Management Spec

## 目的

アーケードゲームのプレイヤー登録と履歴管理を行う、バックエンド寄りの小さな管理アプリを作る。
GitHub Copilot のカスタム指示が、モデル設計、関連設計、API 設計、画面実装、テスト作成にどの程度効くかを確認することが主目的。

## 想定ユーザー

- 店舗運営者
- 管理画面を操作する担当者
- プレイヤーデータを更新するスタッフ

## 何を管理するか

中心に置くデータは次の 4 つに絞る。

- プレイヤー情報
- 所持キャラ
- プレイ履歴
- スコア履歴

表示したい主要項目は次のとおり。

- ユーザー名
- 最終ログイン日
- プレイ回数
- 持っているキャラ
- スコア

## ユーザー体験

担当者は、プレイヤーを検索して詳細を開き、最新の状態を確認し、必要に応じて情報を更新する。
単なる名簿ではなく、最近いつ遊んだか、どのキャラを持っているか、どのくらい遊んでいるかが一目で分かる管理画面にする。

## 画面

最小構成は次の 3 画面。

1. プレイヤー一覧
2. プレイヤー詳細
3. プレイヤー編集

必要なら後から履歴一覧を分ける。

## バックエンドの責務

バックエンドでは次を扱う。

- プレイヤーの CRUD
- 所持キャラの登録と更新
- プレイ履歴の追加
- スコア履歴の追加
- 最終ログイン日の更新
- プレイ回数の集計

## 初版の非対象

初回実装では入れない。

- 認証・認可の厳密な実装
- ランキング機能
- 対戦機能
- 課金機能
- フレンド機能
- 外部 API 連携

## Copilot に最初に投げる依頼文

以下をそのまま Copilot Chat に投げて、実装を始める。

```text
アーケードゲームのプレイヤー登録・履歴管理アプリを作りたいです。

まずは Rails バックエンドから始めてください。
次のデータを扱う前提で、モデル設計、関連、バリデーション、RESTful な API、Serializer、RSpec テストの構成案を出してください。

- プレイヤー情報
- 所持キャラ
- プレイ履歴
- スコア履歴
- ユーザー名
- 最終ログイン日
- プレイ回数
- 持っているキャラ
- スコア

要件にない機能は追加しないでください。
不明点があれば実装前に質問してください。
既存の Rails パターンに合わせ、N+1 を避け、Strong Parameters を使ってください。
```

## この仕様で使う .github

- `copilot-instructions.md`: 全体方針
- `backend.instructions.md`: Rails の設計・命名・関連設計
- `create-rails-model.prompt.md`: モデル作成の起点
- `rails-api.agent.md`: API 設計の相談
- `generate-test.prompt.md`: RSpec の追加
- `docs.instructions.md`: この仕様書や README の追記
- `code-review.prompt.md`: 実装後のレビュー
- `anti-slop/SKILL.md`: 曖昧な命名やテンプレ感を除去

## 実験の見どころ

- 命名が具体になるか
- 関連の切り方が自然か
- 集計や履歴の責務が分かれているか
- README や設計メモの書き方に一貫性が出るか
- 手直しがどれだけ減るか
